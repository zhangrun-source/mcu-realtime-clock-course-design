## 一、开场陈述

按这个顺序说，不要念稿，讲个大概意思即可：

1. **一句话总述**：我做的是基于 8031 单片机的多功能实时钟系统，用 8255A 扩展并行口驱动
   LCD1602 显示，DS1302 做硬件时钟源，24C02 做掉电备份，实现了任务书扩展功能 (1)(2)(5)(7)(10)
   五项，对应评分等级"良好"。

2. **讲清楚三条主线**（这是你方案的核心逻辑，一定要讲清楚"为什么这么分工"）：
   - **8255 是接口核心**：LCD 的数据线走 PA 口、控制线走 PC 口，这是本课程的考核重点。
   - **DS1302 是主时钟**：时分秒、年月日全部由它管，走时精确、断电不丢（靠电池脚）。
   - **定时器只管刷新节拍**：Timer0 每 0.1s（或 0.2s）中断一次，只置一个刷新标志，不做
     计时、不做慢速操作——"中断轻、主循环重"，这是实时系统的标准结构。

3. **一句话带过调试方法论**："我按 8031+8255+LCD → 软件计时 → DS1302 接管 → 按键 →
   24C02 六个阶段递进调试，每步验证通过再往下走，出问题时排查范围很小。"

4. **收尾**："下面我现场演示功能，并可以展示 Proteus 里抓到的关键时序信号。"

## 二、现场演示顺序（照这个走，边做边说）

1. 上电 → 指出第一行 `时:分:秒` + 启动次数、第二行 `20yy-mm-dd Wx`，都是从 DS1302 读的。
2. 按**清除键** → 时间归零并继续走。
3. 按**启停键** → 时间定住；再按一次恢复走。（说明：翻转的是 DS1302 秒寄存器 bit7 CH 位）
4. 按**换挡键** → 光标在时/分/秒间切换，当前位闪烁；按**加数键** → 该位 +1 并按上限回绕。
5. 退出预置（换挡回到 0）→ 时间恢复正常走时。
6. （如果要秀细节）打开 **I2C Debugger** 看 24C02 通信解码，或打开**逻辑分析仪**看 DS1302
   的 SCLK/IO/CE 三线时序，或示波器看 LCD 的 E/RS 波形——指出报告 5.4 节的三张图对应哪里。
7. 断电重启（或复位）→ 指出启动次数 +1、之前的暂停/运行状态被恢复——这是 24C02 掉电备份
   在起作用。

## 三、可能被追问的问题与参考答案

### 硬件相关

**Q: 为什么用 8031 而不是 89C51 之类带片内 ROM 的芯片？EA 为什么必须接地？**
8031 是无片内 ROM 的型号，程序只能放外部程序存储器，所以 EA（外部访问允许）必须接地，
强制 CPU 从外部取指。Proteus 仿真里可以直接把编译好的 hex 挂到 80C31 的 Program File 属性
上，不需要真的接 27C256+74LS373。

**Q: 为什么选 6MHz 晶振而不是常见的 12MHz？**
任务书要求定时器产生 0.1s 定时中断。定时器工作方式 1 是 16 位，最大计数 65536。12MHz 下机器
周期 1μs，0.1s 需要计数 100000，超过 65536，一次装不下，要靠软件二次计数拼凑。改用 6MHz，
机器周期变成 2μs，0.1s 只需计数 50000，一次装入初值 (65536-50000=15536=0x3CB0) 就够，
所以选 6MHz。

**Q: 8255 控制字为什么是 0x80？地址 0x7C00 这些是怎么算出来的？**
控制字最高位 D7=1 表示"方式设定"，其余各位为 0 表示 PA/PB/PC 全部工作在方式0（基本输入
输出）且全部设为输出。地址方面，本设计用 P2 的高位直接做片选和端口选择——`/CS=P2.7`、
`A1=P2.1`、`A0=P2.0`，这样 P0 上的低 8 位地址就不需要 74LS373 锁存了（这是任务书提示的
"省一片锁存器"技巧）。用 `MOVX @DPTR` 访问时只看 DPH（即 P2）的高位，所以 PA 对应
`0x7C00`，PC 对应 `0x7E00`，控制口对应 `0x7F00`。

**Q: 为什么 LCD 的 RW 直接接地？**
只写不读，用固定延时代替"读忙标志位"判断，电路更简单，代价是要保证延时足够长
（清屏/复位类指令 >1.5ms）。

**Q: DS1302 为什么不能外接晶振负载电容？**
DS1302 内部已经集成了负载电容，外面再接电容会让振荡频率偏移甚至不起振，报告里专门提了
这一条。

**Q: 为什么普遍用 32.768kHz 晶振？**
32768 = 2^15，恰好可以用 15 级二分频精确得到 1Hz 秒脉冲，分频电路简单、无余数误差、
功耗又低，这是几乎所有实时钟芯片的标准选择。

**Q: 24C02 的 SCL/SDA 为什么要接上拉电阻？**
IIC 总线是开漏（漏极开路）结构，本身不会主动拉高，必须外部上拉到电源才能形成高电平，
这是 IIC 协议的硬性要求。

### 软件/时序相关

**Q: 定时器中断为什么不直接拿来计时？**
方案一开始确实是软件计时（定时器中断里累加秒分时），这满足任务书"中断服务子程序中
计时"的字面要求，但缺点是断电后时间丢失、精度依赖主晶振。升级到方案二后，改用 DS1302
做时间权威源，定时器中断只保留用来产生"显示刷新"和"闪烁"节拍——既保留了任务书要的
定时中断，又拿到了 DS1302 走时精确、断电不丢的优势。这是我设计里最重要的一个架构决策。

**Q: 启停功能具体怎么实现的？**
DS1302 秒寄存器（读地址 0x81）的 bit7 是 CH（Clock Halt）位，1=停走、0=走时。启停键做的
事就是：读出秒寄存器 → 异或 0x80 翻转这一位 → （解写保护后）写回。因为是异或翻转而不是
直接赋值，所以秒的其余 7 位数值不受影响。

**Q: 为什么要写保护 (0x8E) 解锁/上锁这一步？**
DS1302 的 0x8E 寄存器 bit7 是 WP 写保护位，默认上电后可能是保护状态，写任何时间寄存器前
必须先写 0x00 解锁，写完再写 0x80 上锁防止误写，这是 DS1302 的硬性要求。

**Q: DS1302 和 24C02 的时序有什么区别？（这题几乎必问，报告表 5-3 就是为它准备的）**
两者故意设计得几乎相反，容易搞混：
- DS1302 三线（CE/SCLK/IO），24C02 两线 IIC（SCL/SDA）。
- DS1302 起止靠 CE 拉高/拉低；IIC 靠 SCL 高电平期间 SDA 的跳变（Start/Stop）。
- DS1302 数据**低位先出**（LSB first）；IIC **高位先出**（MSB first）。
- DS1302 没有应答机制；IIC 每字节都有 ACK/NACK 握手。
- DS1302 写完立即可读；24C02 写完要等约 5ms 内部写入周期。

**Q: BCD 码是什么？为什么显示时不能直接当十进制数处理？**
DS1302/24C02 里的时间数据都是 BCD 码——一个字节的高 4 位是十位数字、低 4 位是个位数字
（例如 0x26 表示十进制 26，不是 38）。显示时只需要把高低半字节拆开分别 +'0' 转成 ASCII，
不能再做"除10取余"的十进制换算——那是二次转换，会导致数值放大出错（这正是我踩过的一个
坑，见下面故障分析）。而预置功能需要给时间"+1"运算时，则要反过来：先 BCD→十进制、
加一判断上限、再转回 BCD 写入，因为常规加法不能直接在 BCD 上做。

**Q: 为什么 DS1302_Init（设置初始时间）只在"首次上电"时调用一次？**
DS1302 靠电池脚断电不丢时间，如果每次上电都调用一次 Init，就会把电池保持的当前时间强行
抹回初始值，掉电保持功能形同虚设。所以我用 24C02 里的"魔术字"(0x5A) 判断是不是第一次
上电：没有魔术字 → 第一次，设初值并写入魔术字；有魔术字 → 不是第一次，信任 DS1302
自己保持的时间，不再重设。

**Q: 24C02 为什么不用每秒/每次刷新都写，而是"状态变化时才写"？**
EEPROM 有擦写次数寿命（约10万次），如果每秒轮询写入会很快耗尽寿命。本设计只在按键操作
（清除/启停/预置）改变系统状态后才触发一次备份，兼顾了实时性和器件寿命。

### 调试与排错（这部分是加分重点，体现你真的自己调过）

**Q: 说说你遇到的最有技术含量的一个 bug。**
（这题主动讲、讲透，是报告里明确标出的"答辩很可能被问"的点）
现象：LCD 上时间数字大体正确，但夹杂 `=`、`?` 这类非法字符，而且每次刷新都在变、不固定。
分析思路：数值类错误（比如进制搞错）会**稳定地错**，但这里是**乱跳**，说明是动态的时序
问题而不是静态的逻辑错误。往时序方向查，发现原来的 `DS1302_WByte` 每一位循环结束时
SCLK 停在高电平（`DS_SCLK=1` 收尾），导致命令字节和数据字节之间的时钟相位错位了一拍，
下一次读操作的时钟边沿因此对不上，读回的字节间歇性错位。修复方法是让每一位都以
`SCLK=0` 规整收尾，并且在读字节前把 I/O 先置 1 释放为输入。修完后显示彻底稳定。
**一句话结论**：乱跳 = 时序/动态问题，稳定错 = 逻辑/静态问题，这个判断方法我后续排查
效率高了很多。

**Q: 走时速度快了一倍是怎么回事？**
现象是走时速度大约是实际的两倍，2 倍这个比例正好对应 12MHz/6MHz。原因是 Proteus 里
80C31 器件属性里的 Clock Frequency 设置优先级高于电路图上画的晶振元件，两者没统一，
把两处都改成 6MHz 后恢复正常。

**Q: 显示数值"放大"是怎么回事？**
这就是上面提到的 BCD/十进制混用问题：DS1302 读回的是 BCD 码，如果又用"除10取余"的
十进制拆位公式去显示，相当于对已经是"十位+个位分离"的数据又做了一次十进制换算，
数值直接放大出错。改成直接按半字节拆分（`>>4` 取十位、`&0x0F` 取个位）后正确。

**Q: 预置模式的闪烁一开始为什么不闪，直接消失？**
问题在于闪烁标志的翻转逻辑被错误地放在了显示函数里（显示的时候把标志清零了），而且
中断里当时用的是"赋值"不是"翻转"，导致标志一直保持同一个值。修复思路是明确职责
分离：闪烁标志**只由定时中断**按固定节拍用逻辑取反（`Blink = !Blink`）来翻转，显示函数
**只读取**这个标志决定当前位显示数字还是空格，绝不修改它。这体现的是"状态只有一个
维护源头、其余模块只读"的设计原则。

**Q: 你的分阶段调试是怎么做的，为什么这么做？**
按 ①LCD 显示打通 → ②定时器软件计时 → ③DS1302 接管时间 → ④清除/启停 → ⑤预置(含闪烁)
→ ⑥24C02 备份恢复，六个阶段递进，每个阶段验证通过才进入下一个。好处是出问题时能把
排查范围限定在"最新加的这一块"，不用在全部代码里大海捞针。

## 五、一句话备忘：三个最容易被追问却最容易忘记提的细节

1. 8255 "不锁存低8位地址"是因为用 **P2 高位直接译码**，不是因为 8255 本身不需要地址锁存。
2. DS1302 的 **CH 位**身兼两职：既是"启停"功能的开关，也是预置模式下"防止边设边跳"的
   暂停机制（进入设置先置 CH=1，退出再清 0）。
3. 24C02 备份的是**应用层状态**（运行/暂停、启动次数），不是时间本身——时间由 DS1302
   电池保持，两者分工明确，不要说成"24C02 备份时间"。
